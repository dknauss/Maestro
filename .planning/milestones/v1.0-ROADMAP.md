# Milestone v1.0: WordPress.org Release Readiness

**Status:** ✅ SHIPPED 2026-06-14 (submitted to WordPress.org; awaiting .org review)
**Phases:** 1–5
**Total Plans:** 10 (per STATE.md velocity; per-phase plan counts tracked inline, not as plan files)

## Overview

The plugin was feature-complete and green at intake. This milestone covered everything
needed to publish it responsibly on WordPress.org: confirming the security posture,
finishing the accessibility story to an auditable standard, extending automated test
coverage, producing the .org listing artifacts, and submitting the build to the review
queue. The runtime was git-tagged `v1.0.0` and published as a GitHub Release; the
WordPress.org SVN trunk commit + `1.0.0` tag + asset upload remain pending on external
.org review approval.

## Phases

### Phase 1: Security Review

**Goal**: Every known attack surface in the plugin's REST + storage layer is confirmed safe or hardened
**Depends on**: Nothing (first phase)
**Requirements**: SEC-01, SEC-02, SEC-03, SEC-04, SEC-05
**Status**: ✅ Complete

**Success Criteria** (all TRUE):
1. A test or documented manual check confirms non-capable users and bad nonces are rejected by all REST routes
2. `Config::sanitize_icon()` is confirmed non-executing for all four icon forms (data-URI, URL, dashicon, none) under adversarial input
3. Slug, title, and role inputs are confirmed free of injection and stored-XSS paths via test or static analysis result
4. The `maestro_capability` filter is confirmed incapable of privilege escalation beyond the edit capability — documented in SPEC.md
5. Option writes use `sanitize_*` helpers only, no unserialize of untrusted data, autoload is false — confirmed and documented

**Outcome**: Static Codex Security scan (`317283f_20260614T024544Z`) completed; one low-severity editor DOM XSS hardening issue (`innerHTML` → `textContent` in the shared `el()` helper) found and fixed. REST nonce integration coverage added — missing/invalid nonces now confirmed to reject GET/POST/DELETE `/config` without mutating stored config. No open findings remain.

### Phase 2: Accessibility Audit

**Goal**: The plugin's edit mode is usable without a mouse and announces state correctly to assistive technology
**Depends on**: Phase 1
**Requirements**: A11Y-01, A11Y-02, A11Y-03, A11Y-04, A11Y-05
**Status**: ✅ Complete

**Success Criteria** (all TRUE):
1. All edit operations (select, rename, icon picker, visibility toggle, reset) are reachable and operable by keyboard alone — confirmed by audit
2. Focus lands on the correct element after selecting an item, opening a panel, and opening/closing the icon picker popover
3. Saving triggers a `wp.a11y.speak()` call (or equivalent) so a screen reader announces success or error
4. Icon picker dialog, tabs, and grid have correct ARIA roles and labels — confirmed by audit
5. Keyboard-accessible reordering is documented as a known limitation in readme.txt and/or SPEC.md (full fix deferred to v2)

**Outcome**: Static/code audit closed A11Y-01–A11Y-05. Keyboard item selection (`Enter`/`Space`), focus restoration for popovers, and save success/failure announcements via `wp.a11y.speak()` were added. The v1 keyboard-reordering gap was documented for v2 (later promoted to v1.1 as A11Y-06).

### Phase 3: Verification

**Goal**: Automated tests cover per-role visibility, reset edge cases, and icon sanitization; admin-load overhead and edit-mode payload are measured and acceptable
**Depends on**: Phase 2
**Requirements**: TEST-01, TEST-02, TEST-03, PERF-01, PERF-02
**Status**: ✅ Complete

**Success Criteria** (all TRUE):
1. An e2e test confirms an item hidden from a role is absent for that role's user and present for admin
2. E2e or integration tests cover reset-this-item and reset-all, including edge cases (nothing saved, partial save)
3. Unit tests explicitly cover all four icon forms for both valid acceptance and invalid rejection in `sanitize_icon()`
4. Admin-load overhead (option read + replay) is measured and documented as acceptable (single autoload:false read, O(menu) replay)
5. Edit-mode localized payload size is measured and confirmed only loaded in edit mode (network tab or server-side assertion)

**Outcome**: TEST + PERF combined into one phase for coarse granularity. Unit 44/44; integration 29 tests / 81 assertions (added reset-all idempotence/partial-config coverage plus performance contracts for non-autoloaded storage, edit-mode-only assets, localized payload budget, localized editor labels, and the bundled language path header); Playwright E2E 9/9 (added reset-this-item and per-role visibility).

### Phase 4: Release Assets

**Goal**: All WordPress.org listing artifacts exist and are ready for SVN commit — readme, graphics, screenshots, user docs
**Depends on**: Phase 3
**Requirements**: REL-01, REL-02, REL-03, REL-04
**Status**: ✅ Complete

**Success Criteria** (all TRUE):
1. readme.txt has a `== Screenshots ==` section with numbered entries matching the shipped screenshot files
2. SVN `assets/` directory contains `icon-128x128`, `icon-256x256`, `banner-772x250`, and `banner-1544x500` at correct dimensions
3. Annotated screenshots are captured from the Playground demo and match the `== Screenshots ==` entries
4. User-facing documentation covers: how to enter edit mode, rename, reorder, set icons, configure per-role visibility, and the cosmetic-only caveat

**Outcome**: `.wordpress-org/` contains `icon.svg`, `icon-128x128.png`, `icon-256x256.png`, `banner-772x250.png`, `banner-1544x500.png`, and `screenshot-1.png`–`screenshot-4.png` (PNG dimensions verified with `file`). GitHub `README.md` displays the banner, screenshots, and quick-start; wp.org `readme.txt` carries matching screenshot captions and usage docs; `docs/user-guide.md` holds the longer walkthrough.

### Phase 5: Submit

**Goal**: Plugin Check and WPCS pass on the final build zip and the submission is in WordPress.org review queue
**Depends on**: Phase 4
**Requirements**: REL-05
**Status**: ✅ Complete (submitted; awaiting .org review)

**Success Criteria** (all TRUE):
1. `bin/build.sh` produces a clean zip and `plugin-check` reports 0 errors and 0 warnings against it
2. WPCS (`phpcs`) reports 0 errors and 0 warnings on the plugin source
3. The plugin is submitted to WordPress.org and a confirmation (SVN commit or review ticket) is on record

**Outcome**: `bin/build.sh` produces `build/maestro-menu-editor.zip`; WPCS (`composer lint`) passed 7/7; official Plugin Check 2.0.0 reported no errors on the extracted build zip; `npm audit` reports 0 vulnerabilities after removing the unused `@wordpress/scripts`; local unit 44/44, integration 29/29, E2E 9/9 pass. **Submitted to WordPress.org and in the review queue.** Approval + SVN access are external and pending.

## Milestone Summary

**Decimal Phases:** None — v1.0 ran as a clean 5-phase sequence with no urgent insertions.

**Key Decisions:**
- TEST + PERF combined into Phase 3 (Verification) to keep coarse granularity (5 phases); REL-05 (submit) kept as its own capstone phase.
- DOM XSS hardening: shared editor helper switched from `innerHTML` to `textContent` for localized labels (Codex scan low-severity finding).
- REST nonce gate verified via integration tests simulating the WordPress REST cookie-auth nonce, not merely asserted.
- Localization shipped: `maestro-menu-editor` text domain, POT template + starter catalogs for `es_ES`, `de_DE`, `ja`, `fr_FR`, `pt_BR`, `it_IT`.
- Apply GSD for release-readiness tracking — formalized the path to .org.

**Issues Resolved:**
- Low-severity editor DOM XSS hardening issue closed (`innerHTML` → `textContent`).
- GitHub Dependabot brought to 0 open alerts (8 earlier alerts closed during dependency cleanup, verified 2026-06-14).
- `npm audit` brought to 0 vulnerabilities after removing the unused `@wordpress/scripts` dependency.

**Issues Deferred:**
- Keyboard-accessible reordering (documented v1 limitation) — promoted to v1.1 as A11Y-06.
- Per-item "modified" indicator + discoverable reset — promoted to v1.1 as UX-01.

**Technical Debt Incurred:**
- None blocking. Per-phase plan counts were tracked inline in the roadmap (`/TBD`) rather than as discrete plan files; this project does not use the GSD phase-directory/SUMMARY.md structure.

**External Follow-up (pending, out of our hands):**
- Await WordPress.org review verdict. On approval: SVN commit to `trunk`, tag `1.0.0`, upload `.wordpress-org/` to the SVN `assets/` dir.

**Release anchors:**
- Git tag `v1.0.0` + GitHub Release "Admin Menu Maestro 1.0.0", anchored at the finalize-1.0.0 commit `c5f31b8`.
